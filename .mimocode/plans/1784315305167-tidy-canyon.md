# Destroy & Recreate with Double-Buffering (Strategy A)

## Problem

The WebView pool architecture is the root cause of state-leakage bugs. Parked WebViews carry stale cookies, IndexedDB, `addDocumentStartJavaScript` registrations, and JS global state (`__childFrameGuardInit`) from the previous provider. No amount of cleanup can perfectly reset a parked WebView — it's an endless game of whack-a-mole.

## Solution

**Destroy the old WebView, create a brand new one, cross-fade to hide the latency.**

This solves:
1. `zxcstream ↔ cinemaos` ping-pong (leftover IndexedDB/localStorage nuked by destroy)
2. Guard script staleness (`addDocumentStartJavaScript` is fresh on new WebView)
3. `clearAllState()` race condition (destroying the WebView nukes its entire storage context — no race)
4. Any future state-leak bugs (new WebView = guaranteed clean slate)

---

## Architecture

### Double-Buffer State Machine

```
                    ┌──────────────┐
                    │  IDLE        │
                    │ current = V  │
                    └──────┬───────┘
                           │ sourceUri setter (new provider URL)
                           ▼
                    ┌──────────────┐
                    │  SWAPPING    │
                    │ current = V  │  (still visible)
                    │ incoming = I │  (INVISIBLE, loading)
                    └──────┬───────┘
                           │ onPageFinished OR 4s timeout
                           ▼
                    ┌──────────────┐
                    │  IDLE        │
                    │ current = new│  (incoming promoted)
                    │ old → destroy│
                    └──────────────┘
```

### Key Principle
The old WebView stays VISIBLE while the new one loads in the background. The user never sees a black screen. When the new page is ready, we swap instantly and destroy the old one.

#### C. New `switchProvider(url: String)` Method

This is the core of double-buffering. Called from `sourceUri` setter when `currentWebView` already exists:

```kotlin
fun switchProvider(url: String) {
    val act = appContext.currentActivity ?: return
    if (isSwapping) {
        // Rapid switch — cancel pending swap, destroy incoming, start fresh
        incomingWebView?.let { destroyWebViewCompletely(it) }
        incomingWebView = null
        isSwapping = false
    }

    // 1. Create new WebView INVISIBLE
    val newWv = WebView(act)
    newWv.visibility = View.INVISIBLE
    incomingWebView = newWv
    isSwapping = true

    // 2. Apply settings, clients, JS interface
    applyWebViewSettings(newWv)
    newWv.webViewClient = makeSwapWebViewClient(newWv)
    newWv.webChromeClient = makeWebChromeClient(newWv)
    newWv.addJavascriptInterface(JsBridgeInterface(), "ReactNativeWebView")

    // 3. Apply guard script via addDocumentStartJavaScript
    if (injectedScript.isNotEmpty()) {
        WebViewCompat.addDocumentStartJavaScript(newWv, injectedScript, emptySet())
    }

    // 4. Add to Activity root (behind current — INVISIBLE)
    val rootContent = act.findViewById<ViewGroup>(android.R.id.content)
    rootContent.addView(newWv)
    // Position it at same location as current
    currentWebView?.let { cur ->
        newWv.x = cur.x; newWv.y = cur.y
        val lp = newWv.layoutParams
        if (lp != null) { lp.width = cur.width; lp.height = cur.height; newWv.layoutParams = lp }
    }

    // 5. Start loading
    currentUrl = url
    isLoading = true
    if (referrer.isNotEmpty()) {
        newWv.loadUrl(url, mapOf("Referer" to referrer))
    } else {
        newWv.loadUrl(url)
    }

    // 6. Safety timeout — force swap after 4s
    swapHandler.removeCallbacks(swapRunnable)
    swapHandler.postDelayed(swapRunnable, SWAP_TIMEOUT_MS)

    dispatchEvent("onLoadingStart") { putString("url", url) }
    logRequest("SWITCH", "double-buffer", Uri.parse(url).host ?: "", "navigation", url)
    android.util.Log.d("PlayerWebView",
        "OVERLAY switchProvider url=${url.take(80)} hasScript=${injectedScript.isNotEmpty()}")
}
```

#### D. New `swapViews()` Method

```kotlin
private fun swapViews() {
    if (!isSwapping) return
    swapHandler.removeCallbacks(swapRunnable)
    isSwapping = false

    val incoming = incomingWebView ?: return
    val old = currentWebView

    // Make incoming visible
    incoming.visibility = View.VISIBLE
    incoming.bringToFront()
    syncWebViewFrame()  // Ensure correct position

    // Destroy old WebView completely
    old?.let { destroyWebViewCompletely(it) }

    // Promote
    currentWebView = incoming
    incomingWebView = null

    isLoading = false
    dispatchEvent("onLoadingFinish") { putString("url", currentUrl ?: "") }

    android.util.Log.d("PlayerWebView",
        "OVERLAY swapViews complete url=${currentUrl?.take(80)}")
}
```

#### E. New `destroyWebViewCompletely(wv: WebView)` Method

```kotlin
private fun destroyWebViewCompletely(wv: WebView) {
    wv.onPause()
    wv.loadUrl("about:blank")
    wv.visibility = View.GONE
    val parent = wv.parent as? ViewGroup
    parent?.removeView(wv)
    try { wv.javaClass.getMethod("setWebViewClient", android.webkit.WebViewClient::class.java).invoke(wv, null) } catch (_: Exception) {}
    try { wv.javaClass.getMethod("setWebChromeClient", android.webkit.WebChromeClient::class.java).invoke(wv, null) } catch (_: Exception) {}
    wv.removeJavascriptInterface("ReactNativeWebView")
    wv.destroy()
}
```

#### F. New `makeSwapWebViewClient()` — WebViewClient for Incoming

Same as `makeWebViewClient()` but with swap trigger on `onPageFinished`:

```kotlin
private fun makeSwapWebViewClient(wv: WebView): WebViewClient {
    return object : WebViewClient() {
        override fun onPageStarted(view: WebView?, url: String?, favicon: Bitmap?) {
            currentUrl = url
            isLoading = true
        }
        override fun onPageFinished(view: WebView?, url: String?) {
            // Trigger swap — incoming page is ready
            swapViews()
            // Re-inject guard script on main frame (post-swap)
            if (injectedScript.isNotEmpty()) {
                wv.evaluateJavascript(injectedScript, null)
            }
        }
        override fun shouldOverrideUrlLoading(view: WebView?, request: WebResourceRequest?): Boolean {
            // Same session-locked allowlist check as current WebViewClient
            val url = request?.url?.toString() ?: return false
            val targetHost = Uri.parse(url).host?.lowercase() ?: return false
            if (request.isForMainFrame) {
                val (root, allowed) = synchronized(sessionLock) {
                    lockedRootHost to lockedAllowedHosts
                }
                if (root != null && targetHost !in allowed) {
                    android.util.Log.w("PlayerWebView",
                        "[AB] HIJACK BLOCK (swap WebView): ${url.take(120)}")
                    return true
                }
            }
            return false
        }
        override fun shouldInterceptRequest(view: WebView?, request: WebResourceRequest?): WebResourceResponse? {
            // Delegate to shared ad-blocking logic
            return interceptRequestForWebView(view, request)
        }
    }
}
```

**Note:** The `shouldInterceptRequest` logic is currently inlined in `makeWebViewClient()`. Extract it into a shared `interceptRequestForWebView()` method so both the current and swap WebViewClients use the same blocking rules. This avoids duplicating ~200 lines of ad-blocking logic.

#### G. Modify `ensureWebView()` — First Attach Only

Remove all pool logic. `ensureWebView()` only runs on first attach (when `currentWebView == null`):

```kotlin
private fun ensureWebView() {
    if (currentWebView != null) return
    warmDnsCache()
    val act = appContext.currentActivity ?: return
    WebView.setWebContentsDebuggingEnabled(true)

    val wv = WebView(act)
    applyWebViewSettings(wv)
    wv.webViewClient = makeWebViewClient(wv)
    wv.webChromeClient = makeWebChromeClient(wv)
    wv.addJavascriptInterface(JsBridgeInterface(), "ReactNativeWebView")

    currentWebView = wv
    // Apply deferred props
    userAgent?.let { wv.settings.userAgentString = it }
    if (supportMultipleWindows) wv.settings.setSupportMultipleWindows(true)
    if (javaScriptCanOpenWindowsAutomatically) wv.settings.javaScriptCanOpenWindowsAutomatically = true
    if (injectedScript.isNotEmpty()) {
        WebViewCompat.addDocumentStartJavaScript(wv, injectedScript, emptySet())
    }

    val rootContent = act.findViewById<ViewGroup>(android.R.id.content)
    rootContent.addView(wv)

    pendingLoadUrl?.let { url ->
        pendingLoadUrl = null
        wv.loadUrl(url)
    }
}
```

        try {
            android.os.Handler(android.os.Looper.getMainLooper()).post {
                val warmup = WebView(context)
                warmup.loadUrl("about:blank")

        } catch (_: Exception) {}

## Risk Assessment

- **Double-buffer swap:** Medium complexity. The swap state machine must handle rapid switches (cancel pending swap), detach during swap, and render process gone during swap. All handled by `isSwapping` flag + `destroyWebViewCompletely()`.
- **Chromium warmup:** Low risk. Creates and destroys a dummy WebView at app launch. If it fails, subsequent WebView creation is just slower (~500ms instead of ~50ms).
- **React mountGen removal:** Low risk. The native side now handles provider switches internally. mountGen is still used for crash recovery and manual retry.
- **Shared intercept extraction:** Zero risk. Pure code reorganization, no behavior change.
